Design Review · Ripple MLS Search · April 2026

Four ways to make RelAI feel less like a database,
and more like an assistant.

I read through the desktop app, domain schemas, and ranker. Here's what I found, followed by four design directions for the core search-and-packet flow — each answering a different question about who Ripple is for. Then a short list of focused UX upgrades that are useful in any direction.

Note: I didn't actually boot the stack (no Docker here) — this is grounded in code, not runtime. The mockups use placeholder data + a warm editorial palette; none of it binds you to a final visual language.

What the app already is

Anchors worth designing around — and soft spots worth pulling on.

01 · SHAPE

Client folders are the unit of work

Everything hangs off one client at a time: the search, saves/favorites, extracted criteria, artifacts, and the forthcoming preference embedding. That's a strong organizing primitive most MLS tools don't have. Lean into it.

Anchor
02 · INPUT

Natural language → structured criteria

Agents type a sentence; the backend extracts hard filters, soft preferences, ambiguities, and a confidence. The UI already exposes this — but it reads as a form, not as a conversation with a teammate.

Opportunity
03 · RANKING

Heuristic + semantic, with three states

Each result has price/location/beds/baths/feature/preference sub-scores, plus a semantic state (applied · no-match · unavailable). That's rich explainability the current card barely uses.

Unexploited
04 · OUTPUTS

The real product is a PDF or email

Search is a means; the artifact is the end. Today the flow is: select → click → wait → download. No preview, no framing, no "why these five for Sarah" narrative.

Under-designed
05 · FEEDBACK

save/favorite/hide already encode taste

Three-way actions plus ranked results is a full feedback loop — it's how the folder embedding should get smarter over time. The UI doesn't surface that the system is learning.

Invisible
06 · BRAND

"Pulse" / "Ripple" signals motion

The current UI is a standard shadcn dark template — competent but anonymous. The name suggests something more alive: signals propagating, things coming back. Room for a real voice.

Blank canvas

Four directions

Same app, four different positionings. Each mocks the core screen (one client folder, one search, ranked results). Pick the one that fits who you want Ripple to be for.

Direction A

The Editorial Assistant

Warm paperSerif voiceNarratedCalm density

Ripple talks to you. Results read like a memo from a research associate — "here's what Sarah is looking for, here are the five I'd open with, and here's why." Warm paper palette, Instrument Serif for voice, JetBrains Mono for data. One-at-a-time presentation; depth over breadth.

Differentiating Elevates the agent's judgment Not ideal for 200-result sweeps

The Terminal

Warm paperDense monoKeyboard-firstNarrated

For the high-volume agent who runs 10 searches before lunch. Everything is a keyboard shortcut, everything is comparable at a glance, sub-score sparklines replace prose. Dark background, warm orange highlight — the feel of a Bloomberg terminal for real estate.

Direction C

The Map View

Spatial firstPaper + inkDenseNarrated

Real-estate is inherently spatial — "3 blocks from the park" matters more than "70 on location." A permanent split: criteria + ranked list on the left, a real map with heat-weighted clusters on the right. Hover a listing → pin pulses; click a pin → card detail docks to the map. Brown/cream cartography, not Google blue.

Matches how agents think Makes "near X" tangible Needs real geo data pipeline

Six focused UX upgrades

Small moves that work inside any of the four directions above.

01

Make the match score clickable.

Today the card shows "92 · excellent" with a few tiny bars. You have a full ScoreBreakdown — including the three-state semantic signal. Turn the score into an explainer popover: stacked sub-scores with weights, and a tag showing whether semantic is applied, no-match, or unavailable (these must stay distinguishable — the schema comment says so).

02

Criteria chips as a diff, not a form.

The current CriteriaPanel is a shadcn form. But agents want to see what the extractor heard vs. what they corrected — especially when confidence is low or there are ambiguities. Render criteria as chips with three states: extracted, added-by-agent, removed-by-agent. Ambiguities are inline yellow chips the agent resolves by clicking an option.

03

Folders need a Pulse home.

Right now /folders is a flat list. For an agent with 20 active clients, the question is "who needs me today?" Replace the list with an activity-weighted feed: new listings matching a saved search, stale saves, price drops, packets the client has opened. The product name earns itself.

04

Preview the packet before you generate it.

The PDF is the product. Today it's a black box — click "Generate," wait, download, hope. Show a thumbnail carousel of the actual page layout, with inline toggles: include agent remarks, include photos, cover letter tone, order. Generation stays async; the preview is a styled placeholder until the artifact flips to ready.

05

Make the learning visible and revertible.

Your schema already distinguishes four signal sources by authority (explicit client → agent-tagged → pitched artifact → search soft-pref). Expose them as a ledger per folder: what moved the ranker, why, and a one-click revert. Trust comes from seeing the machine's reasoning, not from hiding it.

06

"Hidden" should narrate, not disappear.

Hide currently removes the row — the count shows "6 hidden" but no way to interrogate. Give hide a reason taxonomy (too-small, wrong-block, tenant-occupied, bad-photos, other) that feeds the folder embedding directly. The ranker gets better signal, the agent gets a conversation-starter with the client: "we hid 6 under $700k because none had parking — should we relax that?"

Where I'd go next

If you give me one of these to chase.

My recommendation — blend A (Editorial) as the visual language with D (Taste profile) as the interaction model, and use C (Map) as a viewmode toggle for geographic queries. Skip B unless your actual users are the high-volume pros who'd value mono density over warmth; it's the most striking direction but also the narrowest.

The six UX upgrades are independent and should land regardless of direction — especially #1 (clickable score), #2 (criteria diff), and #4 (packet preview), all of which are blocked only by design decisions, not infrastructure.

Tell me which direction to deepen (or mix), and whether you want a real interactive prototype (with the actual TanStack + shadcn stack swapped in) or a hi-fi static mock of a single flow end-to-end — search → criteria → rank → packet preview → download.
